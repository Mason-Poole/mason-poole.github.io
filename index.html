Category Two - Algorithms and Data Structures

While my Category One enhancements focused on porting the app to a more accessible medium, for this enhancement, I wanted to add enhancements that would directly improve the user experience compared to the mobile app. In the original mobile app, navigating the list and finding specific items in the database was quite cumbersome, so I decided to focus on this issue for this enhancement. My solution was to implement a merge sort algorithm and a search algorithm into the inventory table, both of which make it significantly easier for users to organize and navigate large databases of items.

To start, when item data is requested from the server, it gets stored as an array of objects on the local client. Both the sorting and filtering functions modify this local item array, a design philosophy chosen to minimize server impact and allow requests to be fulfilled as quickly as possible. The merge sort algorithm is used to sort the local item table by any item property (name, quantity, etc.) in a consistent O(N log N) time.

The filtering algorithm is one of my own designs, which simultaneously filters the database by a substring and orders the results by relevancy, all with an O(N) time complexity. The relevancy ranking sorts items into three categories: first are items where the substring matches the beginning of the name, second are items where the substring matches the start of a word in the name, and last are items where the substring matches at some other point in the name. For example, if the substring for a filter is "an," a sorted list would be "animal crackers, fish and chips, sandwiches."

As for course outcomes, the most important one for this enhancement was the algorithmic principles outcome, which I directly met by developing and implementing both a merge sort and a custom filtering algorithm into the website, solving the problem of large tables being difficult to navigate. Additionally, this work reinforced my use of well-founded techniques, indicated by my usage of algorithms to implement table sorting and filtering, delivering more value to the web application.

Compared to the previous week's enhancements, this category was significantly more straightforward. Thanks to the extensive planning I did before Category One, the only major challenge I faced was implementing the filtering algorithm into the front-end system. While the filtering process was already outlined in my initial enhancement plan, I never thoroughly considered how this filtering algorithm would be used in the system, which led to a design flaw where filtering the local item array couldn't be undone without fetching a new list of items from the back-end, which went against my design philosophy of minimizing server impact. I was able to resolve this issue by drawing the table from a filtered copy of the local item array. However, in the future, it would be better to not only put more thought into how components will be implemented, but also how other components are designed to accommodate all planned features.

Category Three - Databases

The overall goal of my enhancements across the three categories was to port the original Android app to a full-stack web application. While my Category One enhancements ported most of the functionality, one key component was still left: the database. Creating a MongoDB database and linking it to the website through CRUD functions and RESTful API endpoints was the final step in recreating all the core functionality of the original app on the website. When the client application needs to request data from the database, it makes an API request to the server, which processes the request and triggers the appropriate CRUD function to query the database, parses the response, and returns the requested data to the client via the API response. Integrating these concepts into my web application simultaneously improves the project by providing persistent and secure data storage for both user accounts and inventory items, and highlights my ability to integrate said concepts effectively and securely, skills that are crucial in our current data-driven society.

Additionally, I made several security improvements while implementing the database, including API rate limits, password encryption, token-based authentication, and HTTPS routing, making the application significantly more secure. All of these factors combined highlight not only my ability to work with databases and integrate them into complex systems but also my knowledge and prioritization of critical security measures and design patterns, a skill that I believe will only become more crucial as technology evolves.

The designing and implementation of these security measures directly correlate to the security mindset course outcome, which was the only course outcome I had not yet met. With the completion of this category, I had successfully met all five course outcomes through my enhancements.

This final week's enhancements involved the most learning out of any of the three categories. While I had experience developing individual components such as password hashing and user authentication, this was the first time I'd ever integrated all these concepts into a single, complete web application. Completing this week's enhancements has given me a plethora of invaluable hands-on experience integrating these components into a web application, as well as first-hand knowledge on how these components interact with each other in real-world systems. Most of this category's work went smoothly, but one area that posed some challenge was developing the user authentication and authentication system. Of all the components I planned to develop for this category, this was the one I had the least experience with, so I spent a considerable amount of time researching the best authentication methods to ensure my implementation was as effective and secure as possible, as well as testing both manually and through Postman to ensure the system worked as intended and couldn't be easily compromised. I believe all this work paid off, as the IP-based user-token system used in the final enhancement is both effective and secure, ensuring that only logged-in users can access the inventory table and its associated API routes.
